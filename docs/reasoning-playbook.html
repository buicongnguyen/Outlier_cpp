See the proof before writing the patch

Think in contracts, invariants, ownership, and state transitions

Strong C++ answers are not guesses about syntax. They are short proofs: define what must be true, trace the data and state changes, find the first point where the invariant can fail, repair that point, and choose tests that distinguish a correct solution from a plausible one.

Flow of thinking

The universal assessment loop

Do not choose a container, lock, or algorithm until the contract and invariant are explicit.

  1. 1ContractInputs, outputs, ordering, errors, complexity.
  2. 2InvariantWhat must remain true before and after every operation?
  3. 3Owners & stateWho owns bytes? What changes? What is shared?
  4. 4Small traceUse one adversarial input or thread interleaving.
  5. 5RepresentationPick the simplest structure or primitive that proves the invariant.
  6. 6RepairPatch the first broken transition, not the visible symptom.
  7. 7ProofTests, complexity, lifetime, exception, and concurrency assumptions.
If a trace or test fails: return to the invariant or representation. Adding another condition to the symptom usually creates a second bug.

Review task

Build a contract matrix, find a decisive counterexample, rank correctness before performance, then judge clarity.

Debugging task

Symptom → violated invariant → smallest failing trace → root cause → regression test.

Coding task

Contract → state transitions → data structure → edge cases → complexity and exception guarantee.

Flow of data

Ownership and lifetime: follow the bytes

A pointer, reference, iterator, span, or string_view is safe only while its owner keeps the referenced storage valid.

Unsafe borrowed-view flow

  1. Ownership chainCatalog → vector owns string objects → each string owns its character sequence
  2. Observerstring_view points at those characters
  3. Invalidating eventelement mutation/erase/destruction, or vector reallocation, invalidates observed storage
  4. Invalid viewobserver still holds the old address
  5. Later accessundefined behavior or corrupt lookup

Ownership decision

Independent result lifetime?Return an owning value such as vector or string.
Exclusive heap ownership?Use unique_ptr and move it.
Genuinely shared lifetime?Use shared_ptr; break cycles with weak_ptr.
Short borrowed access?Use a reference, pointer, span, or string_view with an explicit lifetime contract.
Ask in order: Who owns the object?When is it destroyed?What can relocate it?Who keeps an observer?
Copy distinction: copying a vector does not invalidate its source. The Catalog failure occurs because a default copy duplicates string_view addresses instead of rebinding them to the copied strings; destroying the source then leaves the copy's views dangling.

State machine

Strong-guarantee transaction pattern: prepare, commit, or roll back

Every observable exit must meet its documented guarantee. The state flow below models the strong guarantee: failure has no observable effect.

Valid old stateAll invariants hold.
PreparePerform work that may allocate or throw.
CommitUse a small non-throwing state change.
Valid new stateAll invariants hold again.
Throw during prepare→RAII cleanup / erase provisional entry→Valid old state

Basic guarantee

After failure, invariants hold and resources are not leaked, but observable state may have changed.

Strong guarantee

Failure has no observable effect: the operation either completes or preserves the old state.

No-throw guarantee

The operation is guaranteed not to fail; destructors and commit primitives often need this property.

Partial update bug

index.insert(key) succeeds → items.push_back throws → index claims an item that does not exist.

Transactional repair

Insert provisional key → try the second step → erase and rethrow on failure → expose success only after both structures agree.

Logic decision + sequence

Atomic check-then-act: make the transition indivisible

Atomic loads and updates do not automatically compose into one transaction.

  1. Load observed statecurrent = balance.load()
  2. Invariant permits change?No → fail without mutation.
  3. CAS succeeds?Yes → this is the linearization point.
  4. CAS fails?current receives the newer value; recompute and loop.

Choose an atomic

One independently meaningful scalar fully represents the invariant, and one load/store/RMW/CAS expresses the transition.

Choose a mutex

Correctness spans multiple values, a container, throwing work, a ledger, or a blocking wait.

Release/acquire publication is a data-flow proof

  1. Producerwrite the payload
  2. Release storeready.store(true, std::memory_order_release)
  3. Synchronizes withready.load(std::memory_order_acquire) reads that released value
  4. Consumerpayload writes now happen-before the read
Ordering heuristic: start with sequential consistency. Relaxed ordering preserves atomicity and the atomic object's modification order, but creates no happens-before edge for surrounding data; it fits the wallet because no payload is published.

Synchronization decision

Can writable sharing be removed?Prefer immutable data, thread ownership, or message passing.
Must an operation wait for work or capacity?Use a mutex-protected predicate with a condition variable, often behind a queue API.
Does one atomic transition prove the complete invariant?Use an atomic load/store/RMW/CAS and state the memory-order proof.
Does correctness span fields, containers, throwing work, or callbacks?Use a mutex and keep the protected invariant together with it.
Measured need for lock-free progress?Only then accept topology, ABA, reclamation, and portability proof costs.

Sequence diagram

Multiple locks: see the wait-for cycle

A deadlock is a graph cycle: each thread owns one resource and waits for the other.

  1. Reject invalid inputamount ≤ 0 or same account
  2. Acquire the setscoped_lock(A, B)
  3. Validate shared statefunds and overflow while both locks are held
  4. Commit both changespreserve the total-balance invariant
  5. RAII unlockevery return path releases both locks

State machine + sequence

Closeable bounded queue: shutdown is part of the contract

The protected state—not the notification—is the truth. Every wait rechecks a predicate while holding the mutex.

  • push: open empty/partial → one more item; notify one consumer.
  • pop: open/closed with data → one fewer item; notify one producer.
  • close: any open state → closed state; notify all producers and consumers.
  • last draining pop: closed draining → closed empty; no reopen transition.
Progress classification: waiting on a mutex or condition variable makes this a blocking queue, not a lock-free or wait-free one.

Algorithm and data pipeline

Separate input occurrences from distinct result rows

For ranked aggregation, preserve every piece of metadata needed by the comparator before reducing the data.

  1. Input · n itemsE2, E1, E2, E1, E3
  2. Aggregatecount + first index per code
  3. u rowsE2:(2,0), E1:(2,1), E3:(1,4)
  4. Composite sortcount ↓, then first index ↑
  5. Take min(k, u)deterministic distinct result

Comparator proof

If counts differ, higher count comes first. Otherwise, lower first-occurrence index comes first.

Complexity proof

Average O(n) aggregation + O(u log u) sorting + O(u) records, with string hashing/copy costs stated separately.

Logic decision tree

Choose evidence that matches the failure

Tools confirm a hypothesis; they do not replace a contract, invariant, or minimal reproducer.

Compile diagnosticRead the first error; reduce templates; verify types and language mode.
Link diagnosticTrace declarations, definitions, ODR, libraries, and CMake target edges.
Wrong resultBuild the smallest counterexample; trace branches and state transitions.
Crash / corruptionUse ASan for out-of-bounds, use-after-free, invalid free, and related memory faults.
Undefined operationUse UBSan for signed overflow, invalid shifts, null/alignment, and similar UB.
Race / intermittent dataUse TSan plus a controlled interleaving and shared-state inventory.
Hang / deadlockTSan primarily detects data races and may report potential lock-order inversions. For an actual hang, use a timeout, all-thread stacks, and a wait-for graph.
Unexpected mutationUse a debugger watchpoint to stop at the write that changes the invariant.
  1. ReproduceMake failure observable.
  2. MinimizeRemove irrelevant code and input.
  3. ClassifyLogic, lifetime, UB, race, hang, build.
  4. InstrumentChoose the matching compiler/tool.
  5. Repair invariantPatch the earliest invalid transition.
  6. RegressAdd the decisive test, then run the full suite.
