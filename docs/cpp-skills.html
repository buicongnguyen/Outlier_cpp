1. C++ Language & Memory Model

What actually happens between your source file and the CPU: value semantics and object lifetime, the features that make modern C++ feel modern (RAII, smart pointers, move semantics, templates, lambdas), and the memory model underneath — stack vs heap, const, and the undefined-behavior contract you sign with the optimizer.

Key terms

Translation & native code
C++ compiles ahead of time straight to machine code: no VM, no JIT, no garbage collector. What you write is what the optimizer turns into instructions — performance and footguns both come from that directness.
Value semantics
Variables hold objects, not references to them: T b = a; copies the whole object (deep, if the copy constructor says so). C++ classes behave like int by default — the opposite of Java/C# class semantics.
Pointer
T* — an address that can be null, reseated, and offset with arithmetic. Owning raw pointers are legacy style; modern code uses raw pointers only as non-owning observers.
Reference
T& — an alias for an existing object: never null, never reseated, no arithmetic. The default way to pass objects you don't want copied: const T& for read, T& for write.
const correctness
const on parameters, members, and methods documents and enforces what cannot change. const T& parameters accept temporaries; const member functions promise not to mutate the object.
Stack vs heap
Automatic (stack) objects die deterministically at scope exit — the backbone of RAII. Heap objects (new) live until deleted; in modern code that delete is done by a smart pointer, never by hand.
RAII
Resource Acquisition Is Initialization: tie a resource (memory, file, lock, socket) to an object's lifetime — acquire in the constructor, release in the destructor. Cleanup then happens automatically on every exit path, including exceptions.
unique_ptr
Sole ownership, zero overhead: exactly one owner, deletes on destruction, movable but not copyable. The default owning pointer — std::make_unique<T>() replaces new.
shared_ptr / weak_ptr
Reference-counted shared ownership: the last owner deletes. Costs an atomic control block. weak_ptr observes without owning — the standard fix for reference cycles that would otherwise leak.
Move semantics
An rvalue reference (T&&) binds to temporaries and to std::move-d objects; the move constructor steals the source's guts (pointers, buffers) instead of copying, leaving it valid-but-unspecified.
std::move
Not a move — a cast to rvalue that permits moving. After moving from an object, don't use its value again (assigning a fresh value is fine).
Copy elision / RVO
Returning a local by value usually copies nothing: the object is constructed directly in the caller's slot. Since C++17 this is mandatory for returned prvalues — so return by value, freely.
Template
Compile-time code generation parameterized by types and values: std::vector<T>, std::sort. Each instantiation compiles a specialized version — zero runtime cost, at the price of compile time and header exposure.
auto / type deduction
auto deduces from the initializer using template rules (drops top-level const and references unless you write auto&/const auto&). Prevents accidental conversions and verbosity.
Lambda
[capture](args){ body } — a compiler-generated function object. Capture by value copies at creation; capture by reference aliases locals (dangling if the lambda outlives them — the classic async bug).
constexpr
Functions and variables evaluable at compile time. Moves work from runtime to the compiler: table generation, size computations, and validation with zero runtime cost.
Undefined behavior (UB)
Contract violations the compiler may assume never happen: out-of-bounds access, signed overflow, dangling dereference, data races. Not "an error at runtime" — the optimizer is allowed to do anything, including appearing to work.
Dangling reference
A pointer/reference/string_view/span to an object that has died: returning a reference to a local, keeping an iterator across a vector reallocation, capturing a local by reference in an escaping lambda.
std::string / string_view
string owns its characters; string_view is a non-owning pointer+length over someone else's characters — free to copy and slice, but it must not outlive the string it views.
std::vector growth
Contiguous array that reallocates when capacity is exceeded — geometric growth gives amortized O(1) push_back, but every reallocation invalidates all iterators, pointers, and references into it.
Initialization
Uninitialized locals of built-in type hold garbage (reading them is UB). Brace initialization T x{...} value-initializes and rejects narrowing conversions; member initializer lists construct members directly.
Exceptions
throw unwinds the stack running destructors (why RAII makes exception safety automatic); catch by const&. Destructors must not throw during unwinding.
lvalue / rvalue
An lvalue is a named, addressable object; an rvalue is a temporary (or something cast to one). Which category an expression has decides what binds to it — rvalues bind to T&& and may be moved from; lvalues need std::move to opt in.
The four named casts
static_cast (checked-at-compile-time conversions), dynamic_cast (checked-at-runtime downcast, needs a polymorphic type), const_cast (add/remove const — using it to actually write is usually UB), reinterpret_cast (bit reinterpretation, last resort). C-style casts silently pick one for you — avoid.
enum class
Scoped enumeration: enumerators live inside the type (Color::red) and do not implicitly convert to int. Fixes the classic unscoped-enum bugs — name collisions and accidental arithmetic.
noexcept
A promise the function does not throw; violating it calls std::terminate. Matters for performance too: vector only moves elements during growth if the move constructor is noexcept — otherwise it copies for safety.
std::optional
A maybe-value without sentinels or heap: optional<int> parse(string_view) makes 'no result' a typed, checkable state instead of -1, nullptr, or an out-parameter flag.
Structured bindings
auto [key, value] : map — unpack pairs, tuples, and aggregate structs into named locals. Removes the .first/.second noise from map iteration.
RTTI
Run-time type information: what typeid and dynamic_cast consult, stored alongside the vtable for polymorphic types. Some codebases disable it (-fno-rtti) for size; frequent dynamic_cast is usually a design smell regardless.

Core concepts

Ownership is the language

Every C++ design question reduces to "who owns this object and how long does it live?" The modern answer is expressed in types: unique_ptr = one owner, shared_ptr = counted owners, raw pointer/reference = borrowed, no ownership. Get this right and there are no leaks and no double frees — without a garbage collector. Get it wrong and you have dangling pointers the compiler won't catch. new/delete in application code is a smell; make_unique/make_shared plus containers cover almost everything.

RAII beats finally

Where other languages bolt cleanup on with try/finally or using blocks, C++ builds it into object lifetime: lock_guard unlocks, fstream closes, unique_ptr deletes — automatically, on every return path and every exception, in reverse declaration order. This is why C++ has no finally: destructors are strictly more general. The discipline: every resource gets a wrapper object; a function that calls a manual release function on three exit paths is a bug generator.

Copies, moves, and when each happens

Pass by value = copy (or move, from a temporary). Pass by const& = no copy, read-only. Return by value = usually free (elision/RVO). std::move turns an lvalue into a movable rvalue — after which the source is hollow. The practical defaults: take sink parameters (things you'll store) by value and move them into place; take read-only inputs by const& (or by value if cheap, like int and string_view); return results by value. Measure before doing anything cleverer.

Templates: compile-time polymorphism

std::sort(v.begin(), v.end(), comp) inlines the comparator and beats qsort — that's templates: the generic code is stamped out and optimized per type, with no dispatch at runtime. Costs: code lives in headers (visible, recompiled per user), error messages historically brutal (C++20 concepts fix this by naming the requirements: template<std::integral T>), and each instantiation adds code size. Use templates for algorithms and containers over unbounded type sets; use virtual functions when the set of types varies at runtime.

UB and the optimizer's contract

The compiler optimizes assuming your program has no UB: no signed overflow, no out-of-bounds, no data races, no dangling access. Violate that and the symptoms are wild — checks deleted, branches folded, "impossible" states. This is why "it works on my machine" means nothing for UB: it can work for years then break on an optimization level bump. Defense in depth: warnings as errors, sanitizers in CI (ASan for memory, UBSan for arithmetic and lifetime, TSan for races), fuzzing on parsers, and value types over pointer juggling.

Views don't own

string_view, span, iterators, ranges, references, and captured-by-reference lambdas all point into someone else's storage. They are fast precisely because they don't copy — and every one of them dangles if the owner dies or reallocates first. The recurring production bug: store a string_view member, or return one built from a temporary std::string, or push_back into the vector you're iterating. Rule: views are for parameters and short-lived locals; anything stored long-term owns its data.

2. Object-Oriented Design

Organizing code so it can change: virtual dispatch and abstract interfaces as C++ expresses them, the special-member-function rules that make ownership safe (rule of zero/three/five), the SOLID principles as day-to-day decisions, and the C++-specific traps — slicing, non-virtual destructors — plus the compile-time alternatives (CRTP, std::function) the language adds to the classic patterns.

Key terms

Encapsulation
An object owns its state and exposes behavior: private data behind member functions, invariants enforced in one place. private/protected/public are the mechanism; friend is the controlled escape hatch.
Inheritance
A class extends a base, reusing and specializing it. C++ allows multiple inheritance; the common sane use is one base class plus abstract interfaces.
virtual / override
virtual opts a member function into dynamic dispatch via the vtable; override in the derived class asks the compiler to verify it really overrides (catching signature typos that would otherwise silently hide).
Pure virtual / abstract class
virtual void f() = 0; — no implementation, class cannot be instantiated. A class of only pure virtuals (plus a virtual destructor) is C++'s interface.
vtable
Per-class table of virtual function pointers; each polymorphic object carries one hidden vptr. A virtual call is an indirect call through it — the (small) price of runtime polymorphism, paid only on virtual functions.
Virtual destructor
Mandatory on any base deleted through a base pointer: without it, delete basePtr runs only the base destructor — the derived part leaks. UB, and a classic interview trap.
Object slicing
Assigning/passing a derived object by value as its base copies only the base part — the derived data and behavior are sliced off. Polymorphism requires references, pointers, or smart pointers.
Rule of zero
The goal: write no destructor, copy, or move operations at all — let members (string, vector, unique_ptr) manage themselves. Classes that own via RAII members compose correctly for free.
Rule of three/five
If you write any of destructor / copy ctor / copy assignment, you almost certainly need all three (manual resource!); with move ctor / move assignment, five. = default and = delete state intent explicitly.
Composition
Building behavior by containing and delegating to other objects ("has-a") instead of inheriting. Swappable, testable, no fragile-base-class coupling.
SRP / OCP
Single responsibility: one reason to change per class. Open/closed: add new behavior as new types plugging into an abstraction, not by editing a switch in tested code.
LSP / ISP / DIP
Substitution: a derived type must honor the base's contract (no strengthened preconditions, no NotSupported throws). Segregation: small client-specific interfaces. Inversion: high-level code depends on abstract interfaces; concretions injected from outside.
Dependency injection
Dependencies arrive through the constructor (reference, or unique_ptr for ownership) instead of being constructed inside. No framework container needed — main() is the composition root.
CRTP
Curiously Recurring Template Pattern: class D : Base<D> — the base calls derived methods resolved at compile time. Static polymorphism: no vtable, fully inlinable, but no runtime substitution.
std::function
Type-erased callable holder: stores any lambda/function/functor with a matching signature. Strategy pattern without a class hierarchy — at the cost of possible allocation and an indirect call.
PIMPL
Pointer-to-implementation: the public header holds only a unique_ptr<Impl>; all members live in the .cpp. Cuts compile-time coupling and keeps ABI stable when internals change.
operator overloading / <=>
Value types define == and (C++20) the spaceship <=> to get all comparisons consistently — often just = default, comparing member by member.
explicit
Blocks single-argument constructors from acting as implicit conversions (void f(Meters m); f(5); compiling silently is how unit bugs are born). Default for single-arg constructors.
Immutability
const objects and members: safe to share across threads (the standard library assumes const = thread-safe reads), safe as map keys, trivially cacheable.
final
On a class: no further inheritance. On a virtual function: no further overriding. Documents design intent and lets the optimizer devirtualize calls it can prove.
Multiple inheritance / diamond
A class may have several bases. When two bases share a grandparent, the grandparent is duplicated unless inherited virtually. Sane convention: inherit one implementation class at most, plus any number of pure interfaces.
NVI idiom
Non-Virtual Interface: public non-virtual methods call private virtual hooks. The base class controls the contract (pre/postconditions, locking, logging) while derived classes customize only the hook.
friend
Grants a named function or class access to private members — for symmetric operators and tightly-coupled pairs (container/iterator). A wide friend list is encapsulation giving up.
Downcasting
dynamic_cast<Derived*>(basePtr) — a runtime-checked walk down the hierarchy (nullptr on failure). Occasional use is fine; a switch on types via dynamic_cast means a virtual function is missing from the interface.
Covariant returns
An override may return Derived*/Derived& where the base returns Base*/Base& — the classic use is clone() returning the concrete type.

Core concepts

Interfaces, C++ style

An interface is an abstract class of pure virtual functions plus a virtual destructor: struct Logger { virtual void log(std::string_view) = 0; virtual ~Logger() = default; };. High-level code takes a Logger& and never mentions FileLogger — that's DIP, and it's what makes the code testable with a fake. The C++ twist: you choose the dispatch mechanism per use — virtual interface (runtime substitution, stable ABI), template parameter (zero overhead, compile-time), or std::function (one callable, maximum flexibility).

Rule of zero first, five when owning

The special member functions (destructor, copy×2, move×2) define what copying and destroying mean for your type. Best case: define none — members like string, vector, unique_ptr already copy/move/destroy correctly, and the compiler composes them (rule of zero). The moment you manage a raw resource — a handle, a buffer, a FILE* — the compiler's defaults are wrong (they copy the handle, then double-free), so you must write or delete all five (rule of five). Declaring any copy/move operation suppresses implicit moves — another reason to stay at zero.

Slicing and the by-value trap

void draw(Shape s) takes a Shape by value: passing a Circle constructs a plain Shape from its base part — radius gone, virtual calls resolve to Shape's. No warning, wrong behavior. Same trap: std::vector<Shape> storing mixed shapes, catching exceptions by value, returning polymorphic objects by value. The rule: polymorphic types are used through Shape&, Shape*, or unique_ptr<Shape>, and are typically made non-copyable or abstract to turn the mistake into a compile error.

Composition over inheritance

Inheritance is the strongest coupling in the language: the child inherits every implementation decision, base changes ripple down, and one hierarchy axis can't express two independent variations (channel × retry policy = subclass explosion). Composition keeps each axis a separate injected component: Sender(Channel&, RetryPolicy&). Inherit for a genuine is-a with a stable contract you dispatch through; contain and delegate for everything else. Note the standard library's own bias: STL containers and algorithms use no inheritance at all — templates and composition throughout.

SOLID as daily decisions

SRP: if describing a class needs "and", split it. OCP: the switch you re-edit for every new case wants to become an interface with one implementation per case — or a variant+visit if the case set is closed. LSP: a derived class that throws "not supported" from an inherited member belongs in a smaller interface (ISP). DIP: OrderService mentions PaymentGateway (abstract), never StripeClient; main() wires the concretions. These are pressure valves, not laws — a three-case switch that never grows is fine.

Patterns, C++ flavored

Strategy is an injected interface — or just a std::function/template parameter when the strategy is one callable. Observer is a vector of callbacks (mind lifetime: who unsubscribes?). Decorator wraps the same interface (BufferedStream around Stream). Factory returns unique_ptr<Base>. Template method inverts to CRTP when you want it resolved at compile time. Visitor is largely superseded by std::variant + std::visit for closed type sets. The C++ question for every pattern: does this variation happen at runtime (virtual) or is it fixed at compile time (template)?

3. Architecture, Toolchain & Ecosystem

Assembling real C++ applications: the compilation and linking model (translation units, headers, the ODR), CMake and package managers, the standard library's containers and algorithms, concurrency with threads and atomics, error-handling strategy, and the everyday toolchain — GoogleTest, sanitizers, clang-tidy, and profilers.

Key terms

Translation unit
One .cpp file after preprocessing (all its #includes pasted in). Compiled independently to an object file; TUs only meet at link time.
Header / include guard
Declarations shared between TUs live in headers; #pragma once (or guards) prevents double-inclusion within one TU. Templates and inline functions must be fully visible in the header.
Declaration vs definition
A declaration announces a name (void f(int);); a definition provides the body/storage. Declare in headers, define once in a .cpp — or mark inline.
ODR
One Definition Rule: every function/variable has exactly one definition across the program (inline/template entities: identical definitions allowed per TU). Violations = linker errors, or silent UB if definitions differ.
Linker
Combines object files and libraries, resolving symbol references to definitions. undefined reference = missing definition/library; note link order matters with static libs on some linkers.
Static vs shared library
Static (.a/.lib): code copied into the executable at link time. Shared (.so/.dll): loaded at runtime, shared between processes, swappable — but now ABI compatibility matters.
ABI
Application Binary Interface: the binary-level contract (layout, mangling, calling convention). Changing a class's members breaks ABI even if the API compiles — why PIMPL and C interfaces guard shared-library boundaries.
CMake
The de-facto meta build system: CMakeLists.txt describes targets and their properties; CMake generates Ninja/Make/VS projects. Modern style: target_link_libraries with usage requirements, not global flags.
vcpkg / Conan
C++ package managers: fetch, build, and wire dependencies into CMake with correct flags/versions — replacing hand-copied source drops and system-wide installs.
GoogleTest / Catch2
The common unit-test frameworks: TEST(Suite, Name), EXPECT_EQ, fixtures, parameterized tests; GoogleMock generates mocks of virtual interfaces. Run via ctest in CI.
Sanitizers
Compiler-instrumented runtime checkers: ASan (heap/stack overflow, use-after-free), UBSan (overflow, bad casts, lifetime), TSan (data races), MSan (uninitialized reads). Cheap insurance — run the test suite under them in CI.
clang-tidy / clang-format
Static analysis (bug patterns, modernization, Core Guidelines checks) and mechanical formatting. Both run in CI so review time goes to design, not style.
Profilers
Measure before optimizing: perf/VTune/Instruments for CPU sampling, heaptrack/ASan for allocations. The hot spot is rarely where you guess.
STL containers
vector (default: contiguous, cache-friendly), array (fixed), deque, list (rarely right), map/set (ordered trees, O(log n)), unordered_map/set (hash, O(1) avg).
Algorithms / ranges
std::sort, find_if, transform, accumulate over iterator pairs — C++20 ranges compose them lazily (views::filter | views::transform). Prefer algorithms to hand-rolled loops.
std::thread / jthread
OS threads; jthread (C++20) joins automatically and carries a stop token. Prefer higher-level std::async/task queues to raw thread management.
Mutex / lock_guard
std::mutex protects shared state; lock_guard/scoped_lock lock RAII-style (scoped_lock takes several mutexes deadlock-free). Data + its mutex belong in one class.
std::atomic
Race-free single-variable operations (counters, flags) without a lock. Default memory order (seq_cst) is correct; weaker orders are an expert tool.
Data race
Two threads touching the same memory, at least one writing, no synchronization: UB — not "stale reads" but license for the optimizer to break you. TSan finds them; mutexes/atomics prevent them.
condition_variable
Sleep until a predicate holds: cv.wait(lock, []{ return !queue.empty(); }) — always with a predicate (spurious wakeups) and under the mutex guarding the state.
Exceptions vs expected
Exceptions for rare, non-local failures (RAII makes propagation safe); std::optional/std::expected (C++23) for expected, local failures — the error is in the return type, visible at every call site.
Warnings as errors
-Wall -Wextra -Werror (or /W4 /WX): the compiler catches shadowing, narrowing, missing returns, unused results. The cheapest static analysis there is.
C++20 modules
import instead of textual #include: compiled once, no macro leakage, faster builds in principle. Toolchain and build-system support is still uneven — headers remain the deployed reality in most codebases.
LTO
Link-Time Optimization: the optimizer sees all TUs together at link, enabling cross-file inlining and dead-code elimination. Slower links, faster binaries; common for release builds (-flto).
Debug vs release builds
Debug: -O0 -g, assertions on — debuggable, slow. Release: -O2/-O3 -DNDEBUG — fast, asserts gone. Bugs that appear only in release are a UB smell; RelWithDebInfo keeps symbols for profiling production-like builds.
std::async / future
Task-level parallelism: auto f = std::async(std::launch::async, work); runs work on another thread; f.get() joins and returns the result (or rethrows its exception). Simpler than raw threads for one-shot tasks.
False sharing
Two threads writing different variables that share a 64-byte cache line ping-pong the line between cores — a silent scalability killer for adjacent counters. Fix: alignas(64) / padding per-thread data.
Microbenchmarking
Google Benchmark and friends: statistically repeated timing of small hot paths. Chief trap: the optimizer deleting the work you meant to measure — use DoNotOptimize, and confirm findings in a real profile.
Symbol visibility
Shared libraries default to exporting everything; -fvisibility=hidden plus explicit export macros shrinks the ABI surface, speeds loading, and keeps internals legally changeable.

Core concepts

The build model explains the errors

Each .cpp compiles alone into an object file; the linker then matches every symbol reference to exactly one definition. That model decodes the error zoo: undefined reference = you declared it but never linked its definition (missing .cpp or library); multiple definition = a function defined in a header included twice (fix: inline or move to a .cpp); include guards don't help across TUs — they only stop double-paste within one. Templates must live in headers precisely because the compiler must see the definition to instantiate it per TU.

Modern CMake is target-based

Think in targets and their requirements, not global variables: add_library(net ...), then target_include_directories(net PUBLIC include) and target_link_libraries(app PRIVATE net). PUBLIC requirements propagate to consumers automatically — no more hand-threading include paths. PRIVATE keeps implementation deps hidden. With vcpkg/Conan providing find_package, a new machine builds with configure + build, no README archaeology. Out-of-source builds keep the tree clean; presets pin compiler/flags for reproducibility; ctest wires tests into CI.

vector until proven otherwise

Container choice is mostly cache behavior: vector is contiguous, so iteration prefetches perfectly — it beats list at nearly everything, including many "insert in the middle" workloads, because pointer-chasing misses cache at every hop. Reach for unordered_map for key lookup (O(1) average, but hashing cost and iteration order chaos), map only when you need sorted order or stable iterators, deque for front+back growth. Then: reserve() when the size is known, and prefer algorithms — std::sort + unique, erase_if — over hand loops with iterator-invalidation traps.

Concurrency that stays correct

The C++ memory model makes unsynchronized shared mutation UB, so the discipline is structural: share nothing by default (thread-owned data, message passing); where sharing is required, pair the data with its mutex in one class so unlocked access is impossible to write; use std::atomic for lone flags/counters. Locks are RAII (scoped_lock), held short, never across callbacks or I/O. Deadlocks come from inconsistent lock ordering — scoped_lock's multi-mutex form fixes the two-lock case. Trust TSan over code review: races hide in "obviously fine" code for years.

Layering that survives frameworks

The clean-architecture shape ports directly: a core library of domain rules that includes nothing from the edges; an application layer defining ports as abstract interfaces (Repository, Clock); infrastructure implementing them with SQLite/HTTP/OS calls; main() wiring it up. In C++ the enforcement is physical — separate CMake targets, and the core target simply doesn't link the infrastructure ones, so the compiler polices the direction. Payoffs: domain tests run in milliseconds with fakes; swapping storage touches one target. The same boundaries also cut compile times — headers stop rippling.

The quality gauntlet in CI

C++ correctness is defense in depth, because the language won't catch lifetime and race bugs for you: warnings-as-errors, clang-format (zero style debate), clang-tidy (bug patterns, modernization), unit tests via ctest, then the same tests under ASan+UBSan and TSan — sanitizers only see code that actually executes, so coverage matters. Fuzz anything that parses untrusted bytes. Release builds add -O2 -DNDEBUG; keep assertions meaningful in debug. This gauntlet is table stakes on serious C++ teams — it's what makes fearless refactoring possible in an unsafe language.

std::mutex + guard:  { std::scoped_lock l(mu);  balance -= x;  log.push(x); }
                     protects INVARIANTS across several operations/objects

condition_variable:  consumer: cv.wait(lock, []{ return !q.empty(); });
                     producer: { lock; q.push(item); }  cv.notify_one();
                     — predicate re-checked after every wakeup (spurious ok)

Further Study

Books

Hands-on drills

Build a small tool end to end: CMake + vcpkg + GoogleTest + sanitizers in CI. Then break it on purpose: leak through a non-virtual destructor, dangle a string_view, race a counter — and watch how ASan/UBSan/TSan each report it.

Reference sites
